## 2026-06-17 · 第 1 次：需求澄清与产品需求说明

### 用户原始需求（节选）

想做一个 threejs-editor 前端工程，技术栈 Vue3 + Three.js。

- 目标：用户通过自然语言描述，生成或修改一个 3D 场景。
- 页面左右结构：
  - 左侧对话区：输入自然语言；系统返回结果；既展示原始 chain-of-thought，又展示「思考摘要 / 执行计划 / 生成步骤 / 修改说明 / 决策理由」等可读内容；支持多轮对话（操场 → +篮球架 → 移动篮球架 → 删除某物）。
  - 右侧结果区三页签：① 3D 预览（旋转/缩放/平移）；② HTML 代码页（Three.js ES Module 导入）；③ DSL 页（可还原场景的 JSON，含元素/灯光/相机/材质/颜色/位置/旋转/缩放/层级；支持多 mesh 拼成复杂物体）。
- 三种 3D 元素来源：① 用户自定义组件（Class → Object3D，如货架，LLM 命中则优先调用）；② 混元等模型生成服务；③ LLM Low Poly 拼接。
- 核心流程：用户输入 → LLM 生成/修改 Three.js 场景代码 → 系统生成/更新 DSL → 右侧预览/HTML/DSL 同步更新。
- **定调：LLM 直接写 3D 代码，DSL 只是后续可能用到的数据结构。**
- 本轮要求：先不写代码、不规划目录、不开始实现；只整理成产品需求说明（核心流程 / 页面功能 / 数据流 / 关键模块 / 多轮修改 / 三者关系 / 风险点）。并把对话记录在项目里。

### AI 的理解与关键建议（摘要）

1. **接受定调并以「代码为真值」展开**：3D 预览=执行代码得到的场景；HTML≈代码+import map 外壳；DSL=运行后场景图的序列化快照（派生、只读）。
2. **DSL 是结果快照、会丢失意图**（如组件函数调用会被展开成 mesh），这是「代码为真值」的必然代价，需提前知情。
3. **多轮修改**：V1 建议「全量重写」+ 用紧凑 DSL 作上下文（而非塞整段代码）省 token；关键前提是给每个物体**稳定语义 id**，移动/删除才成立。
4. **HTML 可移植性是最大隐藏风险**：自定义组件 / 混元模型难以放进「完全独立的 HTML」，需在「内联打包」与「开发态依赖项目模块」之间做选择，产品承诺要和实现对齐。
5. **LLM 执行安全 + Three.js 资源 dispose** 是两个必须做的工程项。
6. 列出 8 个待确认问题，建议先拍板「源真值 / 多轮策略 / HTML 目标 / DSL 是否可编辑」四项。

## 2026-06-17 · 第 2 次：第 1 阶段实现（最小可运行骨架）

### 用户要求（节选）

搭建 Vue3 + Three.js 最小可运行前端页面，只做基础布局，不实现完整 LLM 能力。要求：TS、npm、代码结构清晰可靠、字段类型严谨。左右分栏；左侧对话区（消息列表 + 输入框 + 发送按钮 + 本地 mock 回复，含思考摘要/执行计划/生成说明，**不用 "chain-of-thought" 字段名、不展示原始思考**）；右侧三页签（3D 预览 / HTML 代码 / DSL），3D 预览展示最简立方体，HTML/DSL 先放 mock。明确不做：真实 LLM、混元、自定义组件注册、复杂 DSL 解析。

### AI 实现与关键决策（摘要）

- **选型**：Vite 6 + Vue 3.5（`<script setup lang="ts">`）+ Three.js 0.171 + @types/three；TS 严格模式（strict / noUnusedLocals / noUnusedParameters）。
- **类型层**：消息用**判别联合** `ChatMessage = UserMessage | AssistantMessage`；系统消息三字段 `summary / plan / generationNote`，严格避开 CoT；页签用 `ResultTab` 字面量联合；DSL 用 `DslSnapshot` 结构化类型（含相机/灯光/对象，含 `Vec3`）。
- **3D 预览**：`SimpleScene` 类，职责单一，`mount()` / `dispose()`，dispose 回收 geometry/material/renderer/controls；ResizeObserver 自适应；用 `v-show` 保持场景常驻避免反复重建。
- **OrbitControls** 从 `three/examples/jsm/controls/OrbitControls.js` 导入（保证 @types/three 类型可解析）。
- **布局**：CSS 变量 `--split-left: 40%` 控制左右比例。

### 过程中遇到并修复的问题

1. `ResizeObserver.observe(this.container)` 报 null 不可赋值 → 改为局部 const 收窄。
2. `vite.config.ts` 用 `node:url` 缺 node 类型 → 安装 `@types/node` 并在 tsconfig.node.json 加 `"types": ["node"]`。
3. dev server 端口 5173 被占用 → 验证时改用 5180。

### 验证结果

- `npm run type-check`：通过，无报错。
- `npm run build`：成功（48 模块，Three.js 致产物约 559KB / gzip 150KB，属正常）。
- `npm run dev`：dev server 正常启动，`/` 返回 HTTP 200，`/src/main.ts` 模块转换正常。

## 2026-06-17 · 第 4 次：第 3 阶段（DSL 格式设计，仅设计不写代码）

### 用户要求（节选）

设计 DSL 数据结构与示例，暂不实现解析器。能力：还原整个场景；描述 scene/camera/lights/objects/materials/geometries/transform/children/metadata；对象类型 primitive/group/component/externalModel/lowPolyComposite；复杂物体由多 parts 组成；每对象稳定 id 便于多轮增改删移；LLM 友好、可转 Three.js、可转 HTML。输出：设计说明+字段含义+5 个示例（立方体/low-poly 小车/component 货架/externalModel/操场）+多轮按 id 修改方案。

### AI 设计要点（摘要）

- **顶层**：`version / metadata / scene / camera / lights / materials(可选) / objects`。
- **层级用统一 `children`**：group 与 lowPolyComposite 的「parts」都是 children，类型只表达意图。转换 Three.js 几乎零成本。
- **id 规则（多轮根基）**：全局唯一、稳定（**不把层级路径写进 id**，否则移动会破坏稳定）、语义 snake_case、重写时保留已有 id。
- **对象判别联合**：primitive(group+material) / group / lowPolyComposite / component(component+props) / externalModel(format+url)。
- **旋转用度数**（XYZ，渲染转弧度）、颜色 hex、向量数组、可选字段带默认值——全为 LLM 友好。
- **多轮两种模式**：V1 整文档重写（保留 id）；V2 按 id 操作补丁（add/update 深合并/delete/replace/reparent）。
- 给出 5 个完整 JSON 示例 + 操场→+篮球架→左移→删看台 的三轮 operations 端到端示例 + 校验规则 + 转换说明。

## 2026-06-17 · 第 5 次：DSL 设计修订（确认 5 项决策 + 纠正因果关系）

### 用户决策

1. parts 统一用 `children` —— 认可。
2. **因果关系纠正（重要）**：DSL 后续根据 **3D 场景构建完成后生成**，**LLM 不修改 DSL 来影响 3D 场景**。3D 场景是因，DSL 是果。AI 上一版误把"按 id 操作补丁修改 DSL"当成编辑通路，已纠正。
3. 材质先**只内联**，不做顶层共享材质注册表。
4. `rotation` 用**度数**。
5. **同时给 `rotation`(Euler 度数) 和 `quaternion`**，并存时 quaternion 优先。

### AI 据此修订 02-DSL设计.md

- 新增「§1 数据流与因果关系」：明确编辑回路 `用户输入→LLM改3D场景→构建Object3D→序列化生成DSL`，LLM 不编辑 DSL；还原回路仅用于从保存 DSL 重建场景。
- **移除 operations 操作补丁**（上一版的错误设计）。

- 新增 `quaternion` 字段（`[x,y,z,w]`，与 rotation 二选一，并存优先）；去掉顶层 materials 注册表，材质只内联。

## 2026-06-18 · 第 6 次：确立 Code-first 权威架构（仅架构，不写代码）

### 用户核心修正

产品**不是 DSL-first**，而是 **Code-first**：
1. Three.js 代码是源数据；
2. 3D 预览由运行 Three.js 代码产生；
3. DSL 从运行后的 Three.js 场景对象中**提取**；
4. LLM 多轮主要修改 Three.js 代码；
5. DSL 可作上下文辅助 LLM 理解场景，但**不当渲染源**；
6. 不实现"由 DSL 渲染场景"主流程；
7. 不设计成"改 DSL 影响场景"。
要求整理：整体数据流、三者因果关系、LLM 职责、前端职责、DSL 提取来源、为何不由 DSL 驱动、风险点、LLM 代码约束。不写代码、不规划目录、不实现。

### AI 整理要点（详见 03 文档）

- **数据流**：单轮 `输入→组装Prompt→LLM输出Three.js代码→前端①HTML②沙箱运行得Object3D渲染③从场景提取DSL→存上下文`；多轮每轮重写代码、重运行、重提取。
- **因果关系**：代码=SSOT(根)；预览=代码的视觉投影(执行)；DSL=代码的结构投影(提取)。编辑回路单向 `代码→场景→DSL`；DSL→场景 仅用于还原旁路。
- **LLM 职责**：理解意图/场景、决策元素来源、生成改 Three.js 代码、维持稳定 id、输出结构化可读内容；不编辑 DSL。
- **前端职责**：编排每轮、应用代码(HTML/预览/DSL)、错误处理与自修复、资源 dispose、上下文管理、异步加载处理；不反向渲染 DSL。
- **DSL 提取来源**：从**运行后的 Object3D 场景图**遍历序列化；**不**静态解析代码文本、**不**问 LLM 要 DSL（保证与场景必然一致）。
- **为何不 DSL 驱动**：代码表达力远超 DSL（动画/过程化/着色器/交互）、LLM 更擅长写代码、避免双源真相、省构建层、往返保真、符合初始定位。
- **风险**：代码执行安全、运行报错、id 纪律靠 LLM、整段重写成本与回归、DSL 有损、组件/模型归属丢失(需 userData 标记)、异步加载、资源泄漏、确定性、上下文膨胀、HTML 可移植性。
- **LLM 代码约束（契约）**：每对象带稳定 `userData.id`；用 userData 标记 lowPolyComposite/component/externalModel；结构反映层级；用标准 geometry/material；变换走 Object3D transform 不烘焙；多轮保留既有 id；提供确定执行入口。
- 另列执行契约(§9)、待确认 4 项(执行入口形态/多轮上下文传什么/沙箱方案/有损内容策略)。

## 2026-06-18 · 第 7 次：第 5 步——LLM 代码生成规范（仅规范，不写代码）

### 用户要求（节选）

设计 LLM 生成 Three.js 场景代码的约束规范，使 DSL 可稳定提取。要点：scene/camera/renderer/lights/objects 五件套；每对象稳定 id；每个 Object3D/Mesh/Group 写 userData(id/name)；复杂物体用 Group 包、Group 与每个 child 都标注；材质/颜色/变换须可从 Object3D/Material 读取；自定义组件与外部模型也补 userData；不用遍历不到的隐藏状态；代码可被后续 LLM 改；多轮保留 id 不重命名；**修改已有场景时基于上一轮代码增量修改而非整段重写**。

### AI 设计要点（详见 04 文档）

- **userData 规范**：通用 `id/name/type`；type 枚举 primitive/group/lowPolyComposite/component/externalModel/light/camera；按类型附加字段(component+props / format+url / lightType / target)；可选 metadata/source/dslIgnore。
- **标注方式**：Mesh 标 primitive、几何/材质实时读不重复；Group 标复合类型；Light/Camera 也标 id（camera 显式存 target）；给出"标注→DSL 落点"映射表。
- **low-poly 组织**：一逻辑物体=一 Group(type=lowPolyComposite)，part 为 Mesh 子节点各自标注，变换走 Object3D 不烘焙，part id 用 `<父id>_<角色>` 稳定描述名。
- **component**：调用方在根补 userData(type/component/props)；组件工厂应自标内部 parts(接 id 前缀)。
- **externalModel**：加载后在根补 userData(format/url)；异步须用宿主可 await 的加载方式，pending 标记。
- **id 稳定**：持久对象 id 跨轮不变、新增分配新 id、删除即移除、按 id 定位改删移。
- **可改性(支持增量)**：一对象一块、块首 `// @id <id>` 注释、变量名=id、低耦合、参数上提、稳定顺序；附增量编辑风险一句(每轮运行校验+失败回退重发该块)。
- **反模式清单 13 条**：不写 id / 烘焙变换 / 隐藏状态 / 自定义几何着色器不标注 / 合并多 part 进一 Mesh / InstancedMesh 表示命名对象 / 只 lookAt 不存 target / 随机无种子 / 整段重写 / 跨轮重命名 id / 动态增删后立即提取 / 匿名 Group / 组件模型不标注。
- 一致性：把 03 §8 的 dslType/modelUrl 统一为 type/url。
- **关键决策记录**：用户选"增量修改"而非整段重写（与 03 曾建议的 V1 整段重写相反），AI 按用户决定设计并加一句风险提示。
- 待确认 4 项：执行暴露机制 / 组件 parts 标注责任方 / InstancedMesh 是否需专门 DSL 节点 / 提取时机。
