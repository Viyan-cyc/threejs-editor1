# Three.js 场景代码生成规范（LLM 必须遵守）

> 状态：**仅规范设计**，不写实现、不规划目录。
> 背景：Code-first 架构（见 [03-架构设计-Code-first.md](./03-架构设计-Code-first.md)）。前端运行 LLM 生成的 Three.js 代码，再**从运行后的场景对象树提取 DSL**。本规范约束 LLM 生成的代码，使 DSL 提取**稳定、可读、保真**。
> 关联：[02-DSL设计.md](./02-DSL设计.md)（DSL 格式）。

---

## 0. 一句话

**代码必须自描述**：每一个可提取的对象都把"我是谁、我是什么、我的可读属性"写进 `userData` 与标准 Three.js 属性里，让遍历器能无歧义地读出来。可提取的状态**全部落在场景树上**，不藏在闭包/模块变量里。

---

## 1. 总体代码规范

1. **完整五件套**：代码必须构建明确的 `scene`、`camera`、`renderer`、`lights`、`objects`。
2. **可独立运行**：代码作为脚本可直接在浏览器跑（配 import map 即为 HTML 代码页内容）；渲染循环正常工作。
3. **以函数返回暴露场景（已决策）**：代码写成**构建函数** `createScene(THREE, ctx)`，返回 `{ scene, camera, renderer }`；前端调用它拿到引用，供提取器遍历。`ctx` 由前端注入（画布/容器、`host`、就绪信号等）。HTML 代码页 = 该函数 + 一小段启动调用，照常可独立运行。详见 [03](./03-架构设计-Code-first.md) §9。
4. **状态全在树上**：所有要被 DSL 记录的状态（对象、变换、材质、层级）都必须体现在场景图的 Object3D 上，**禁止用闭包变量、模块级变量、外部 Map 等场景遍历看不到的地方**承载这些状态（见 §9 反模式）。
5. **可复现**：涉及随机时用固定种子，保证多次运行结果一致。
6. **可改性**：代码组织要便于后续 LLM 增量修改（见 §8）。

## 2. Object3D.userData 字段规范

### 2.1 通用字段（每个可提取对象必填）

| 字段 | 类型 | 含义 |
|---|---|---|
| `id` | string | **稳定、全局唯一、语义化**的标识（snake_case；见 §7） |
| `name` | string | 人类可读显示名 |
| `type` | enum | 该对象的 DSL 类型，取值见下 |

`type` 取值（与 [02](./02-DSL设计.md) 对象类型对齐，另加 light/camera 用于标注）：
`'primitive' | 'group' | 'lowPolyComposite' | 'component' | 'externalModel' | 'light' | 'camera'`

> 同时建议把 `object.name` 也设为 `id`，便于 three 开发者工具/调试识别（与 `userData.name` 并存：前者=身份，后者=显示名）。

### 2.2 按类型的附加字段

| type | 附加 userData 字段 | 说明 |
|---|---|---|
| `primitive` | 无（geometry/material 从 Mesh 实时读） | 颜色/金属度/粗糙度等从 Material 读；变换从 Object3D 读 |
| `group` | 无 | 仅作容器 |
| `lowPolyComposite` | 无（parts 即其 children） | 父 Group 标此类型；children 各自标注 primitive |
| `component` | `component`: 组件名；`props`: 对象 | 如 `{ component:'Shelf', props:{levels:4,...} }` |
| `externalModel` | `format`: `'glb'|'gltf'`；`url`: 字符串 | 如 `{ format:'glb', url:'https://.../m.glb' }` |
| `light` | `lightType`: `'ambient'|'hemisphere'|'directional'|'point'|'spot'` | 强度/颜色/位置从 Light 实时读 |
| `camera` | `target`: `[x,y,z]` | 相机的注视点（three 相机本身不持久化 lookAt，故显式存） |

### 2.3 可选通用字段

| 字段 | 含义 |
|---|---|
| `metadata` | 任意附加信息对象 |
| `source` | 来源标记：`'llm' | 'component' | 'loader'` |
| `dslIgnore` | `true` 表示**排除出 DSL**（临时辅助、gizmo 等） |

## 3. 各对象标注方式

### 3.1 Mesh（primitive）
```js
mesh.userData = { id: 'cube_1', name: '立方体', type: 'primitive' }
mesh.name = 'cube_1'
```
- 几何与材质**不重复写进 userData**，提取器直接读 `mesh.geometry` / `mesh.material`。
- 必须用标准 geometry（Box/Sphere/Cylinder/Cone/Plane/Torus/Ring…）与 MeshStandardMaterial，提取器才能映射回 DSL；自定义 BufferGeometry/ShaderMaterial 会被记为"未知/通用"（有损，应避免或标注）。

### 3.2 Group（group / lowPolyComposite / component / externalModel 的容器）
```js
group.userData = { id: 'car_1', name: '小汽车', type: 'lowPolyComposite' }
```
- 复杂物体的**根**用 Group，并按语义标 `type`。
- 每个子节点各自有完整 userData（见 §4）。

### 3.3 Light
```js
light.userData = { id: 'sun_1', name: '太阳光', type: 'light', lightType: 'directional' }
```
- 灯光也是可识别对象，**必须有 id**。
- intensity/color/position 从 Light 对象实时读；提取器据此填 DSL `lights[]`。

### 3.4 Camera
```js
camera.userData = { id: 'camera_main', name: '主相机', type: 'camera', target: [0, 0.5, 0] }
```
- position/fov 从相机实时读；`target` 必须显式存（不要只靠 `camera.lookAt()`，因其不持久）。

### 3.5 场景根 / 场景级属性
- `scene.background`、`scene.fog` 由提取器直接读，无需 userData。
- 参考网格（GridHelper）若需进 DSL，按 light 同理给 userData，或用 `dslIgnore:true` 排除。

### 3.6 标注 → DSL 字段映射（摘要）

| 场景对象 | userData / 属性 | DSL 落点 |
|---|---|---|
| Mesh | type='primitive' + geometry/material + transform | `objects[]` primitive |
| Group | type='lowPolyComposite'/'group' + children | `objects[]` 对应类型，children |
| 组件根 | type='component' + component + props | `objects[]` component |
| 模型根 | type='externalModel' + format + url | `objects[]` externalModel |
| Light | type='light' + lightType + 强度/色/位 | `lights[]` |
| Camera | type='camera' + target + position/fov | `camera` |

## 4. low-poly 复杂物体的代码组织规则

1. **一个逻辑物体 = 一个 `THREE.Group`**，根节点标 `type:'lowPolyComposite'`，带 id/name。
2. **每个 part = 根 Group 的一个 Mesh 子节点**，各自标 `type:'primitive'` + id/name。
3. **part 的变换写在 Mesh 自身**（position/rotation/scale），**禁止烘焙进几何顶点或矩阵**。
4. **整体摆位用根 Group 的 transform**；part 用局部坐标（相对父级）。
5. **part 用标准 geometry + 标准材质**，便于映射回 DSL。
6. **id 命名**：part 的 id 用 `<父id>_<角色>` 的固定描述名（如 `car_1_wheel_fl`）。这是**稳定的描述性名字**，不随移动改变——合规（id 规则禁止的是"按运行时父级动态拼路径"，不是禁止描述性前缀）。

---

## 5. component 对象的标注规则

1. 组件（用户的 Class/工厂）产出一个 Object3D。**调用方（LLM 代码）在拿到对象后，必须在根上补 userData**：
   ```js
   const shelf = new Shelf({ levels: 4, width: 2 })
   shelf.userData = { id: 'shelf_1', name: 'A区货架', type: 'component', component: 'Shelf', props: { levels: 4, width: 2 } }
   ```
2. **parts 标注由 LLM 补齐（已决策）**：组件本身**不自标**内部。LLM 在 `new Shelf(...)` 拿到对象后，**遍历其子树**，按 `<组件id>_part_<n>`（或语义角色，如 `shelf_1_board_2`）为每个子件补 `userData`（id/name/type='primitive'）。这样 DSL 能展示组件内部结构。
3. 若 LLM 未补齐，DSL 该节点只记 component 根（有损），应尽量避免。
4. 组件的对外 props 必须如实写进 `userData.props`，供还原与分析。

---

## 6. externalModel 对象的标注规则

1. 用加载器（如 GLTFLoader）加载；**加载完成后，在模型根（`gltf.scene`）上补 userData**：
   ```js
   loader.load(url, (gltf) => {
     const root = gltf.scene
     root.userData = { id: 'statue_1', name: '雕塑', type: 'externalModel', format: 'glb', url }
     scene.add(root)
   })
   ```
2. **异步处理**：模型加载是异步的。代码必须使用前端提供的、**可被宿主 await 的加载方式**（或暴露加载完成信号），确保提取器在加载完成后才遍历；未完成时该对象标记为 pending。
3. 模型整体变换（position/scale/rotation）写在根上。
4. 模型内部 mesh 一般不再逐一标注（根标记足够）；如需结构细节可标注。

---

## 7. 多轮修改时的 id 稳定规则

1. **持久对象 id 不变**：跨轮修改时，已存在对象的 `userData.id` **必须保留**，禁止重命名。
2. **新增对象分配新 id**：与已有 id 不冲突；同类追加数字后缀（`car_2`）。
3. **删除即移除**：被删对象的 id 直接消失，不要把该 id 复用到别的对象。
4. **基于上下文定位**：LLM 每轮拿到"上一轮代码 + 当前 DSL 快照（含 id 清单）"，按 id 找到目标对象做改/删/移。
5. **拆分/合并**时审慎分配稳定 id，保证语义连续。

---

## 8. 代码可改性规范（支持增量修改）

为让 LLM 能**基于上一轮代码做增量修改**（而非整段重写），代码须这样组织：

1. **一对象一块、块头标记 id**：每个对象用独立代码块，块首注释 `// @id <id>`，便于 LLM 精确定位与替换。
   ```js
   // @id car_1
   const car_1 = new THREE.Group()
   car_1.userData = { id: 'car_1', name: '小汽车', type: 'lowPolyComposite' }
   // @id car_1_body
   const car_1_body = new THREE.Mesh(new THREE.BoxGeometry(2.2,0.5,1), ...)
   car_1_body.userData = { id: 'car_1_body', name: '车身', type: 'primitive' }
   car_1.add(car_1_body)
   scene.add(car_1)
   ```
2. **变量名与 id 一致**：变量名直接用 id（如 `car_1`），减少认知负担与改错。
3. **低耦合**：对象之间避免隐式依赖；改一个对象尽量不动其它块。
4. **参数上提**：颜色、尺寸等用顶部常量或对象参数表达，便于局部修改。
5. **稳定构建顺序**：对象按一致顺序声明，避免重排引起大 diff。
6. **避免炫技**：不用难读的链式/单行写法；保持平铺直叙。

> 增量编辑风险（一句话）：LLM 增量改代码比整段重写更易出错；每轮改完应做运行校验，失败时回退让 LLM 重发该块（或该块对应的完整代码）。

## 9. 不推荐写法清单（反模式）

| # | 反模式 | 原因 |
|---|---|---|
| 1 | 对象不写 `userData.id` | 提取器无法稳定识别，DSL id 不稳 |
| 2 | 把变换烘焙进几何顶点/矩阵 | 提取器读不到 transform |
| 3 | 用闭包/模块变量/外部 Map 承载场景状态 | 场景遍历看不到，DSL 漏掉（违反"状态全在树上"） |
| 4 | 用自定义 BufferGeometry/ShaderMaterial 又不标注 | DSL 有损，无法映射 |
| 5 | 把多个逻辑 part 合并进一个 Mesh | 丢失结构，无法单独 id/编辑 |
| 6 | 用 InstancedMesh / 程序化批量却不标专门节点 | 必须用 `instances` 专门 DSL 节点表达（见 [02](./02-DSL设计.md)），否则 DSL 无法记录批量对象 |
| 7 | 只用 `camera.lookAt()` 不存 target | target 不持久，相机目标丢失 |
| 8 | 随机不设种子 | 不可复现，重运行场景变化 |
| 9 | 每轮整段重写代码 | 违反增量修改要求，token 贵、易回归 |
| 10 | 跨轮重命名已有 id | 破坏多轮定位与 DSL 稳定 |
| 11 | 运行期动态增删对象后立即提取 | DSL 是提取时刻快照，会漏掉/多出瞬时对象；保持提取时机一致 |
| 12 | 深层匿名 Group 不标 id | 层级不可读、不可定位 |
| 13 | 组件/模型根不补 userData | DSL 无法识别类型，降级为散 mesh |

## 10. 已确认决策（2026-06-18）

- **执行暴露机制 = 函数返回**：代码写为构建函数 `createScene(THREE, ctx)` 返回 `{ scene, camera, renderer }`。→ §1 第 3 条、[03](./03-架构设计-Code-first.md) §9
- **组件 parts 标注由 LLM 调用后遍历补齐**（组件本身不自标）。→ §5
- **InstancedMesh / 程序化大批量对象用专门 DSL 节点 `instances`**（不禁止；schema 待细化）。→ §9 第 6 条、[02](./02-DSL设计.md)
- **DSL 提取时机为指定时刻**：由"代码就绪信号"（`host.ready()` 或入口返回 ready Promise）驱动，而非固定首帧。→ [03](./03-架构设计-Code-first.md) §5

> 本规范 4 项待定均已确认。仍待设计：`instances` 节点的完整 schema。
