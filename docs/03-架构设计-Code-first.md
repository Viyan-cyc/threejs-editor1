# Three.js Editor · 架构设计：Code-first（权威方向）

> 状态：**仅架构理解与建议**，不写代码、不规划目录、不实现功能。
> 本文档确立产品的权威架构方向，优先级高于此前任何"DSL 驱动"的措辞。
> 关联：[01-产品需求说明.md](./01-产品需求说明.md)、[02-DSL设计.md](./02-DSL设计.md)。

---

## 0. 一句话

**Three.js 代码是唯一源数据。3D 预览由运行这份代码产生；DSL 从运行后的场景对象中提取。LLM 多轮修改的是 Three.js 代码，不是 DSL。DSL 永远不驱动 3D 场景。**

---

## 1. 整体数据流

### 单轮
```
用户自然语言输入
   │
   ▼
[前端] 组装 Prompt = 系统指令 + 组件目录 + 当前场景上下文(代码 + DSL快照) + 历史
   │
   ▼
[LLM] 结构化输出 = { 思考摘要, 执行计划, 生成说明, 决策理由, ★ Three.js 代码 ★ }
   │
   ▼
[前端] 拿到 Three.js 代码后，三件事：
   ① HTML 代码页  ← 代码本身(+ import map 外壳)
   ② 3D 预览      ← 在沙箱里【运行代码】→ 得到 Object3D 场景 → 渲染
   ③ DSL 页       ← 从【运行后的场景对象】中【提取】→ JSON
   │
   ▼
[前端] 保存场景状态(当前代码 + DSL 快照) → 作为下一轮上下文
```

### 多轮
每轮重复上述流程。**LLM 拿到「上一轮的 Three.js 代码（+ DSL 快照辅助理解）+ 新指令」，输出新的/修改后的 Three.js 代码**。前端重新运行、重新渲染、重新提取 DSL。

---

## 2. Three.js 代码 / 3D 预览 / DSL 三者因果关系

三者**不是平级**，是一条单向因果链：

| 角色 | 本质 | 因果位置 |
|---|---|---|
| **Three.js 代码** | 唯一源数据（SSOT） | **根**：一切的起点 |
| **3D 预览** | 代码的「视觉投影」（经执行） | 代码 →（执行）→ 场景 →（渲染）→ 预览 |
| **DSL** | 代码的「结构投影」（经提取） | 代码 →（执行）→ 场景 →（提取）→ DSL |

- **编辑回路严格单向**：`代码 → 场景 → DSL`。代码改了，预览和 DSL 都跟着重算。
- **DSL → 场景** 这条路**只用于"从保存的快照还原场景"**，是可选的旁路，**不是主流程**。
- 预览和 DSL 都是代码的"派生视图"：预览 = 执行后的视觉呈现；DSL = 执行后的结构描述。两者都**可从代码重新生成**，本身不持有独立权威。

> 类比：代码是源文件，预览是"运行效果"，DSL 是"运行后自动生成的结构文档"。改文档不会改程序，改程序才会改文档。

---

## 3. LLM 在多轮对话中的职责

1. **理解用户意图**（要生成什么 / 改什么 / 删什么 / 移什么）。
2. **理解当前场景**：借助上下文里的 **DSL 快照**（紧凑结构）+ 上一轮代码，知道现在场景里有什么、各自 id。
3. **决策元素来源**：自定义组件 / 外部模型 / Low Poly 拼接，并给出**决策理由**。
4. **生成/修改 Three.js 代码**：输出的代码运行后应当得到目标场景。
5. **维持稳定 id**：跨轮重写代码时，**保留已有对象的 id**，仅给新对象分配新 id（见 §8 约束）。
6. **输出结构化可读内容**：思考摘要 / 执行计划 / 生成说明 / 决策理由（不含原始思考）。

**LLM 明确不做的**：直接编辑 DSL、把 DSL 当场景定义输出、通过 DSL 影响场景。

---

## 4. 前端在多轮对话中的职责

1. **编排每一轮**：组装 Prompt → 调 LLM → 解析结构化输出。
2. **应用代码**：
   - 更新 HTML 代码页（代码 + import map 外壳）。
   - **在沙箱里运行代码** → 得到新 Object3D 场景 → 渲染到 3D 预览。
   - **从运行后的场景对象提取 DSL** → 更新 DSL 页。
3. **错误处理**：代码运行报错时 try/catch 捕获，必要时把错误回灌给 LLM 自修复，并保留上一份可用场景。
4. **资源管理**：每轮重建场景前，dispose 旧场景的 geometry/material/texture，防 GPU 泄漏。
5. **上下文管理**：维护对话历史 + 当前场景状态（代码 + DSL 快照），控制上下文长度（摘要/截断）。
6. **异步处理**：外部模型加载是异步的，DSL 提取需等加载完成（或标记 pending）。

**前端明确不做的**：把 DSL 反向渲染成场景（主流程）、把用户对 DSL 的编辑同步回场景。

---

## 5. DSL 应该从哪里提取

**从「运行后的 Object3D 场景图」中提取**——即遍历代码执行后产生的活场景对象，序列化为 DSL。

**三个"不从"：**
- **不从代码文本静态解析**：代码是命令式的，可能有循环、过程化生成、组件调用、加载器——静态分析无法得到真实结果。运行后的场景才是 ground truth。
- **不向 LLM 索要 DSL**：LLM 可能"说一套做一套"，它写的代码实际产生的场景与其口述 DSL 不符。从运行结果提取，才能保证 DSL 与场景**必然一致**。
- **不让 DSL 反向成为输入**：DSL 是输出，不参与驱动。

**提取器 = 场景图序列化器**：遍历 Object3D 树，读每个节点的类型 / transform / geometry / material / `userData` / children，按 [02-DSL设计.md](./02-DSL设计.md) 的 schema 产出 JSON。

**提取时机（已决策：指定时刻）**：提取**不在固定首帧触发**，而是等"代码就绪信号"——生成代码调用 `host.ready()`，或执行入口返回 ready Promise（外部模型加载完成后再 resolve）。这样动态 / 异步场景也能在稳定状态下提取。

> 对 `component` / `externalModel` 这类对象：运行后场景里只剩它们产生的 mesh。要能在 DSL 里还原成"这是个货架组件/外部模型"而非"一堆散盒子"，**必须靠对象身上的 `userData` 标记**（见 §8）。这是提取保真度的关键契约。

---

## 6. 为什么不应该由 DSL 驱动 3D 场景

1. **代码的表达力远超 DSL**：Three.js 代码能做动画、过程化生成、循环、数学运算、自定义着色器、运行时交互——静态 DSL 全都捕捉不了。若 DSL 驱动渲染，产品就退化成一个有限的"积木拼装器"，丢掉 Three.js 的全部能力。
2. **扬长避短**：LLM **极擅长写代码**，写严格结构化数据（精确枚举、合法参数）则容易出错。Code-first 让 LLM 做它最擅长的事。
3. **避免双源真相打架**：若代码和 DSL 都能驱动场景，两者必然产生分歧，一致性维护成本巨大。单一源（代码）最简单可靠。
4. **省掉一个大构建层**：DSL→场景 需要为每个 DSL 特性写一套渲染器/代码生成——巨大的维护面。代码→场景 只是"运行它"。
5. **往返保真**：从运行结果提取的 DSL 天然与场景一致；手写/LLM 直出的 DSL 不保证。
6. **符合最初的产品定位**："LLM 直接写 3D 代码"是从第一天就确定的方向。

**代价（承认）**：代码不如 DSL 易做"外科手术式"局部编辑（所以多轮倾向整段重写）；干净提取 DSL 需要约束 LLM（稳定 id + userData 标记）。这些都可控。

## 7. 风险点

| # | 风险 | 缓解 |
|---|---|---|
| 1 | **运行 LLM 生成代码的安全风险** | 沙箱隔离（iframe sandbox / Web Worker），收窄可用 API，CSP |
| 2 | **LLM 代码运行报错** | try/catch 包裹，捕获错误回灌 LLM 自修复，保留上一份可用场景 |
| 3 | **稳定 id 纪律全靠 LLM 自觉** | Prompt 强约束 + 校验；提取器对无 id 对象兜底补 id（但语义稳定性弱） |
| 4 | **整段重写的成本与回归** | 大场景 token 贵、可能漏改既有物体 → 用 DSL 作紧凑上下文、限定改动范围 |
| 5 | **DSL 提取有损** | 过程化内容/自定义着色器/动画无法静态序列化 → 接受有损，DSL 只记"结构"不记"行为"，文档说明 |
| 6 | **组件/外部模型归属丢失** | 运行后只剩 mesh → 组件工厂与加载器**必须**打 userData 标记（契约） |
| 7 | **外部模型异步加载** | DSL 提取需等加载完成，处理好竞态与 pending 状态 |
| 8 | **重建场景的资源泄漏** | 每轮 dispose geometry/material/texture |
| 9 | **确定性** | 同 prompt 可能生成不同代码/场景；如需"重试同款"需固定随机种子 |
| 10 | **代码/上下文膨胀** | 多轮后代码累积 → 摘要旧轮，只保留最新代码 + DSL 快照 |
| 11 | **HTML 可移植性**（PRD 已列） | 组件/模型在"独立 HTML"里需内联或引用，产品承诺要对齐 |

## 8. 为让 DSL 提取稳定，LLM 生成代码须遵守的约束（契约）

提取器遍历活场景图。要让 DSL **稳定、可读、保真**，生成代码必须遵守：

1. **每个逻辑对象必须带稳定 `userData.id`**
   - 创建代表一个场景元素的对象时，设置 `obj.userData.id = '<语义 snake_case>'`。
   - 全局唯一、跨轮稳定（同一对象 id 不变）、语义化、**不编码层级路径**。
   - 这是第一契约。缺了它，提取器只能补任意 id，跨轮不稳定。

2. **用 `userData` 标记非 primitive 的对象类型**
   - 低模组合：父 Group 设 `userData.type = 'lowPolyComposite'`。
   - 组件：组件产生的根设 `userData.type = 'component'` + `userData.component = 'Shelf'` + `userData.props = {...}`。
   - 外部模型：加载后的根设 `userData.type = 'externalModel'` + `userData.format` + `userData.url`。
   - 否则提取器按默认（primitive/group）处理。

3. **场景结构要反映 DSL 层级**
   - 一个逻辑物体 = 一个 Group 包住它的 parts，Group 带组合 id/标记。
   - 不要把本属于同一物体的 mesh 散放成平级。

4. **用标准可识别的 geometry / material**
   - 用 BoxGeometry/SphereGeometry/… 与 MeshStandardMaterial(color/roughness/metalness)，提取器能映射回 DSL。
   - 自定义 BufferGeometry / ShaderMaterial 会被序列化为"未知/通用"（有损，可接受但应标注）。

5. **变换用 Object3D 自身的 transform**
   - 位置/旋转/缩放写在对象的 position/rotation/scale 上，**不要烘焙进几何顶点或矩阵**，否则提取器读不到。

6. **旋转单位**：代码里写 Three.js 原生的弧度即可；提取器负责弧度→度数转换。代码侧无需特殊约束。

7. **多轮重写时保留既有 id**：持久对象的 `userData.id` 不变；只给新增对象分配新 id。

8. **提供确定的执行入口**（见 §9）：代码要有前端可调用的、确定的方式产出场景根。

## 9. 代码执行契约（前端如何运行 LLM 代码并拿到场景）

为让前端能"运行代码 → 拿到 Object3D 场景根 → 提取 DSL"，生成代码需满足一个**执行契约**（具体形式实现时定，这里只描述要求）：

- 代码需暴露一个**确定的入口**，供前端调用并获得场景根 Object3D。
- **入口形态（已决策：函数返回）**：代码定义为构建函数 `createScene(THREE, ctx)`，返回 `{ scene, camera, renderer }`；前端在沙箱里调用它，拿到 scene 加入预览、交给提取器。`ctx` 由前端注入（画布/容器、`host`、就绪信号 `ready()` 等）。
- 入口需**幂等可重建**：同一份代码运行多次结果一致（涉及随机时用固定种子）。
- 代码不应保留阻止 dispose 的外部引用。

> 这个契约同时决定了"HTML 代码页"展示什么（就是这份代码 + import map 外壳），保证预览页与 HTML 页是同一份代码的两种呈现。
