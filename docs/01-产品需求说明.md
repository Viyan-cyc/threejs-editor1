# Three.js Editor 产品需求说明（v0.1）

> 状态：需求理解阶段，未开始实现。
> 来源：2026-06-17 首次需求沟通。本文不写代码、不规划目录结构，只梳理需求与建议。
>
> ⚠️ **架构方向以 [03-架构设计-Code-first.md](./03-架构设计-Code-first.md) 为权威**：Three.js 代码是唯一源数据，3D 预览由运行代码产生，DSL 从运行后的场景对象提取；LLM 多轮改代码、不改 DSL；DSL 不驱动场景。本文相关表述以此为准。

---

## 0. 一句话定位

一个「自然语言 → 3D 场景」的 Web 编辑器：用户用自然语言描述需求，LLM **直接生成 / 修改 Three.js 场景代码**，右侧实时同步 **3D 预览 / 可复制的 HTML 代码 / 可还原场景的 JSON DSL**。

技术栈：Vue 3 + Three.js。

---

## 1. 核心目标

1. 自然语言驱动创建与修改 3D 场景。
2. 支持多轮对话**增量修改**已有场景（增加、改位置、删除等）。
3. 三种元素来源统一接入：自定义组件 / 模型生成服务（混元等）/ LLM Low Poly 拼接。
4. 三视图（预览 / HTML / DSL）始终一致、实时同步。
5. 对话区既展示原始 chain-of-thought，又展示「思考摘要 / 执行计划 / 生成步骤 / 修改说明 / 决策理由」等可读内容。

---

## 2. 核心用户流程

**单轮：**
1. 用户在左侧输入需求（例：「生成一个操场」）。
2. 系统组装 LLM 输入：系统指令 + 组件目录 + 当前场景状态 + 对话历史 + 用户输入。
3. LLM 返回**结构化**结果：原始思考 + 思考摘要/执行计划/生成步骤/修改说明/决策理由 + Three.js 场景代码。
4. 系统应用代码：执行得到新场景 → 同步刷新预览、HTML、DSL。
5. 左侧渲染可读结构化内容 + 可折叠的原始 CoT。

**多轮：** 每轮把「当前场景状态」作为上下文带入，LLM 在其基础上生成新代码。

---

## 3. 页面功能

整体左右结构。

### 左侧 · 对话区
- 输入框（多行 + 发送）。
- 消息流：用户消息 + 系统消息。
- 系统消息分层：
  - **可读层（默认展示）**：思考摘要、执行计划、生成步骤、修改说明、决策理由。
  - **原始层（折叠 / 可展开）**：完整 chain-of-thought。
- 多轮历史保留与滚动；后续可支持重试 / 编辑某轮。

### 右侧 · 结果区（三页签）
1. **3D 预览**：Three.js 场景；支持旋转 / 缩放 / 平移；场景重建时平滑替换。
2. **HTML 代码**：当前场景对应的完整可运行 HTML（import map + ES Module 导入 three）；代码高亮 + 复制。
3. **DSL**：可还原场景的 JSON；描述元素、灯光、相机、材质、颜色、transform（位置/旋转/缩放）、层级、组合（多 mesh 拼成复杂物体）；支持折叠 / 复制。

---

## 4. 数据流

```
用户输入
   │
   ▼
[Prompt 组装]  ← 组件目录 + 当前场景状态(DSL快照) + 历史
   │
   ▼
[LLM]  → 结构化输出 { CoT, 摘要, 计划, 步骤, 修改说明, 理由, 场景代码 }
   │
   ▼
[代码应用层]
   ├──► 3D 预览：在沙箱中执行代码 → 新 Object3D 场景
   ├──► HTML 页：代码 + HTML 模板(import map) 拼装
   └──► DSL   ：遍历运行后场景图序列化为 JSON（派生快照）
   │
   ▼
[场景状态保存] → 作为下一轮上下文
```

**关键点：** 代码是「可运行的真值」，DSL 是「运行结果的派生快照」（符合「DSL 只是后续可能用到的数据结构」这一定调）。

---

## 5. 关键模块

1. **对话 / 会话管理**：消息历史、多轮上下文、场景状态挂载。
2. **Prompt 组装器**：系统指令 + 组件目录 + 场景状态。
3. **LLM 适配层**：调用模型、解析结构化输出、错误重试。
4. **代码执行 / 预览引擎**：沙箱执行 Three.js 代码、场景重建、资源回收（dispose）。
5. **场景序列化器**：Object3D 树 → DSL JSON。
6. **HTML 生成器**：代码 → 可独立运行的 HTML。
7. **组件目录 / 注册表**：自定义组件清单 + 自然语言描述（供 LLM 选择）。
8. **模型生成接入**：混元等服务（异步、带 loading）。
9. **三视图同步**：预览 / HTML / DSL 一致性管理。

---

## 6. 多轮对话如何修改已有场景（核心难点）

示例链：操场 → +篮球架 → 移动篮球架 → 删除某物。

要做「移动 / 删除」，LLM 必须能**识别并定位**目标物体。在「LLM 直接写代码」模型下，三种可行做法：

- **(A) 全量重写**：每轮 LLM 拿到「当前场景状态 + 新指令」，输出新完整代码；靠自身保持命名一致（如 Group 名 `basketballHoop`），「移动 / 删除」就是改自己写的代码。**最简单、最稳，V1 首选。** 缺点：大场景 token 贵，可能误改既有部分。
- **(B) 增量补丁**：LLM 输出对 DSL / 代码的 add/update/delete 操作（靠稳定 id）。省 token、精准；但要求稳定 id 体系 + LLM 能精确操作，复杂度高。
- **(C) DSL 作为工作记忆**：每轮把紧凑 DSL（而非冗长代码）作为场景状态喂给 LLM，LLM 输出新代码；DSL 兼做增量编辑目标。

**建议：** V1 采用 (A) 全量重写；上下文里用 **DSL 快照（紧凑）** 描述当前场景，而不是塞整段代码，以省 token、减少误改。中后期场景变大后再演进到 (B)/(C)。

无论哪种，都要给每个物体一个**稳定的语义 id**（如 `basketballHoop_1`）——这是「移动 / 删除」能成立的基础。

---

## 7. HTML代码 / DSL / 3D预览 三者关系（核心决策）

已定调：**LLM 直接写 3D 代码，DSL 是派生数据结构。** 据此三者关系为：

- **3D 预览 = 真值来源**：执行代码得到运行时场景。
- **HTML 代码 ≈ 代码本身**：代码 + 一层 import map / 模板外壳，目标是「复制出去能独立运行」。
- **DSL = 运行后场景图的序列化快照**：遍历 Object3D 树生成，用于「还原场景」。

由此带来的重要取舍（需提前知情）：

- DSL 是「结果快照」，会**丢失意图**。例如 LLM 调用了 `createShelf()` 组件函数，DSL 里只剩展开后的若干 mesh，看不到「这是个货架组件」。对「还原渲染」足够，对「按意图再编辑」有损。
- 若将来想做「改 DSL → 反向影响代码 / 场景」，就需要双向同步，复杂度跳一档。**建议先保持单向**（代码 → 场景 → DSL），明确 DSL 只读。
- 自定义组件 / 混元模型在「独立 HTML」里的可移植性见第 9 节风险。

---

## 8. 三种 3D 元素来源

| 来源 | 触发 | 代码里的形态 | 对 HTML / DSL 的影响 |
|---|---|---|---|
| ① 自定义组件（Class → Object3D，如货架） | LLM 判断命中组件目录 | `import { Shelf } from '...'; new Shelf(...)` | HTML 需带入该组件代码；DSL 是展开后的 mesh |
| ② 模型生成服务（混元等） | LLM 判断需要精细模型 | 异步生成 → 加载 glTF | HTML 需引用模型资源 URL；DSL 记录模型引用 |
| ③ LLM Low Poly 拼接 | 兜底 / 通用 | 基础 geometry 组合（多 mesh 可成 Group） | 完全自包含，HTML / DSL 最干净 |

- 需要 LLM 显式给出**选择理由**（决策理由字段），尤其是「为什么选组件而非 Low Poly」。
- 多 mesh 组合物体：DSL 用 Group / children 表达层级，并给组合体一个语义 id。

---

## 9. 风险点与建议

1. **源真值要一锤定音**：代码 vs DSL 谁是真值影响全局。按偏好定为「代码」，但接受 DSL 有损、只读。
2. **LLM 代码执行安全**：浏览器跑 LLM 生成的代码 → 用 iframe sandbox 或 Web Worker 隔离；代码可能报错 → try/catch + 把错误回灌给 LLM 自修复。
3. **Three.js 资源回收**：每轮重建场景必须 dispose geometry / material / texture，否则 GPU 内存泄漏。
4. **全量重写的退化**：大场景下 LLM 可能漏改 / 误改既有物体 → 用紧凑 DSL 作上下文 + 关键物体稳定 id 缓解。
5. **自定义组件 / 模型的 HTML 可移植性**：独立 HTML 难以包含外部组件代码或大模型资源 → 要么内联打包，要么明确「HTML 为开发态、依赖项目模块」，**产品承诺要和实现对齐**。
6. **组件目录契约**：LLM 怎么知道有哪些组件？需要一份带自然语言描述的组件清单注入 prompt。
7. **模型生成异步与耗时**：混元生成慢 → loading / 占位 / 失败重试 UX。
8. **坐标与单位约定**：「左侧」相对什么？需要固定的相机朝向 / 坐标系约定，否则 LLM 理解不一致。
9. **确定性 / 可复现**：同一 prompt 可能生成不同场景；如需「重试同款」要固定随机种子。
10. **DSL schema 演进**：DSL 会变 → 加版本号字段。
11. **多轮上下文长度**：历史 + 场景状态会膨胀 → 摘要 / 截断策略。

---

## 10. 待你确认的关键问题

1. 源真值确认走「代码为真值、DSL 只读派生」吗？（影响全部后续设计）
2. V1 多轮策略选「全量重写」吗？
3. 「HTML 代码页」目标是**完全独立可运行**（内联一切），还是**依赖项目模块的开发态代码**？（决定自定义组件 / 模型怎么处理）
4. DSL 是否需要可编辑并反向生效？还是永远只读？
5. ① 自定义组件的注入方式：你会提供组件清单（名称 + 描述 + 入参）吗？
6. ② 混元模型生成：产物格式（glTF？）、托管方式（URL / 本地）？
7. 三个页签是「同一场景的三个视图」（强一致），对吗？
8. 是否需要「导出 / 保存场景」（持久化 DSL 或 HTML 到文件）？
